# Sprint 1 Report — Главная страница со списком записей и фильтром по статусу

**Dates:** 2026-05-12 — 2026-05-14
**Status:** Complete

## Goal

Реализовать главную страницу с компонентом списка записей и фильтром по статусу как базовую поверхность для будущих CRUD-операций.

## What was built

- **U-1 (список записей).** Создан `RecordList`, читает записи через `loadRecords()` из `storage.js`, рендерит каждую запись с заголовком и статусом-бейджем. Записи отсортированы по `createdAt` (новые сверху).
- **U-2 (фильтр по статусу).** Создан `StatusFilter` с тремя кнопками. Активная подсвечена через Tailwind-классы `bg-blue-500 text-white`. Фильтрация делается на главной странице через `useMemo`, не внутри `RecordList`.
- **U-3 (empty state).** Если после фильтрации массив пустой — `RecordList` рендерит блок с текстом из `copy.emptyList`. Отдельная ветка для случая «вообще нет записей» и «нет записей с этим фильтром» — обе используют один и тот же текст. Возможно, стоит развести — отметил в Notes.

Все тексты UI вынесены в `src/lib/copy.js`, в JSX строк нет.

Unit-тесты добавлены: 4 кейса на `RecordList` (рендер списка, пустой массив, сортировка, выбор записи) и 3 кейса на `StatusFilter` (рендер, выделение активной кнопки, вызов onChange). Все зелёные.

## ADR Compliance

- **ADR-001 (no backend):** ✅ Сетевых запросов в новом коде нет, только чтение из localStorage.
- **ADR-002 (schema записи):** ✅ Не менял структуру записи. Использую только существующие поля.
- **ADR-003 (UI-тексты через copy.js):** ✅ В JSX нет строк, все через `copy.[ключ]`. Добавил ключ `pageTitle` для заголовка страницы — не был в скоупе, но без него заголовок остался бы хардкодом. Отметил в Notes.

## Known Issues

- Сортировка по `createdAt` сделана в `home.jsx` внутри `useMemo` вместе с фильтрацией. По мере роста списков это может стать узким местом — пока 5000+ записей не предполагаются, неактуально.
- Empty state не различает «нет записей вообще» и «нет записей под фильтром». Один и тот же текст. Возможно, в Sprint 2 стоит развести.
