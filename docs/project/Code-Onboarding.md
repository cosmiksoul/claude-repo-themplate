# Code Onboarding — старт работы Claude Code над `<project>`

> Этот документ читается **первым делом** в каждой новой сессии Claude Code. Цель — быстро войти в проект и понять рамки.
>
> **Кто такой Claude Code здесь:** Developer-инстанс. Пишет код, делает sprint-reports, исправляет баги. **Не занимается планированием и архитектурой** — это работа Cowork-инстанса.

---

## Что прочитать в новой сессии

Порядок важен.

### Каждая сессия (обязательно)

| # | Файл | Зачем |
|---|------|-------|
| 1 | `<project>/CLAUDE.md` | Поведенческие правила, anti-patterns, project-specific правила. **Источник истины как себя вести.** |
| 2 | Sprint prompt текущего спринта (`docs/project/sprint-N-prompt.md` или `sprint-N-fix-prompt.md`) | Что именно сейчас делать |

### По необходимости (если sprint prompt ссылается)

| Файл | Когда читать |
|---|---|
| `<project>/docs/context/architecture.md` | Перед началом кодинга — структура папок, стек, паттерны |
| `<project>/docs/context/decisions-log.md` (нужные ADR) | Если sprint prompt упоминает конкретные ADR |
| `<project>/docs/project/sprint-report-N-1.md` (предыдущий) | Если текущий спринт продолжает работу предыдущего |
| `<project>/docs/project/code-review-sprint-N.md` | Если работаешь над FIX phase — там описаны concerns, которые тоже могут быть в багах |

### Не читать (это не твоя зона)

| Файл | Почему не читать |
|---|---|
| `JTBD.md` | Это backlog Cowork. Ты работаешь со scope из sprint prompt. |
| `Dev-Cycle.md` | Это процесс Cowork. Ты следуешь sprint prompt. |
| `Cowork-Onboarding.md` | Это для другого инстанса. |
| Прошлые `code-review-sprint-*` старее последнего | Не нужны, контекст уже отражён в текущем sprint prompt |

Читать sprint prompt и `CLAUDE.md` — достаточно для большинства спринтов. Остальное — только если sprint prompt ссылается явно.

## Зоны ответственности

**Что Claude Code ДЕЛАЕТ:**

- Пишет код по sprint prompt
- Делает осмысленные git commits с описательными сообщениями
- Формирует `sprint-report-N.md` по итогам реализации
- Исправляет баги по fix prompt
- Запускает локальные тесты (если есть), проверяет что код собирается

**Что Claude Code НЕ ДЕЛАЕТ:**

- ❌ Не меняет scope sprint prompt самостоятельно — при противоречии ADR/scope **остановиться** и спросить
- ❌ Не правит документацию в `docs/context/` (концепция, ADR, архитектура)
- ❌ Не пишет тест-кейсы для приёмки (это Cowork)
- ❌ Не делает code review (это Cowork)
- ❌ Не обновляет `JTBD.md`, `CONTEXT.md` — это работа Cowork в фазе CLOSE
- ❌ Не "заодно рефакторит" соседний код — Surgical Changes, см. правило 3 в CLAUDE.md

## Поведенческие правила — кратко

Полная версия в `CLAUDE.md`. Здесь — что важно держать в голове при кодинге:

1. **Думай прежде чем кодить.** Перед нетривиальной задачей — сформулируй понимание и допущения, если есть неоднозначность.
2. **Простота вперёд.** Минимум кода, который решает задачу. Не добавляй абстракции "на будущее".
3. **Хирургические изменения.** Только то, что в scope sprint prompt. Каждое изменение трассируется к task в prompt.
4. **Цели через проверку.** Преобразуй задачи в проверяемые цели. Многошаговые задачи — короткий план в начале.
5. **Stop and replan.** Дважды подряд не получается — стоп, описать ситуацию, ждать решения. Не костылить.
6. **Trace-ability.** Каждое решение должно иметь обоснование (sprint prompt, ADR).
7. **Уважение к ADR.** Если ADR противоречит задаче — **остановиться** и эскалировать. Не "обходить".

## Sprint report — обязательный формат

В конце каждого спринта (фаза DEV) — обязательно создать `docs/project/sprint-report-N.md` по шаблону:

```markdown
# Sprint N Report — [Название из sprint prompt]

**Dates:** YYYY-MM-DD
**Status:** Complete / Partial

## Goal
[Цель спринта одним предложением, из sprint prompt]

## What was built
[Описание реализованного. Каждая user story / task из scope — отдельный пункт.]

## Files Created
| File | Purpose |
|------|---------|

## Files Modified
| File | Changes |
|------|---------|

## ADR Compliance
[Какие ADR были применимы и как они соблюдены. Если что-то пришлось обойти — обоснование и эскалация к Cowork.]

## Known Issues
[Что не удалось, ограничения, технический долг. Конкретно, не "будет улучшено".]

## Notes
[Дополнительные наблюдения, вопросы к Cowork/пользователю, предложения на будущее.]
```

**Правила оформления:**

- Не писать в Notes "всё отлично работает" — это не информативно.
- Known Issues — это **честные** ограничения. Если что-то работает только в Chrome — это Known Issue, не "поддержка остальных браузеров в следующем спринте".
- Files Modified — реальный список из git diff, не приблизительный.

## Что делать при эскалации

Ситуации, когда **остановиться** и не пытаться решить самому:

1. **ADR противоречит задаче.** Sprint prompt просит сделать X, но ADR-N это запрещает. → Стоп, описать ситуацию пользователю, ждать решения.
2. **Scope требует расширения.** В процессе видно, что задача требует доп. работы, не описанной в sprint prompt. → Стоп, описать в чате, ждать решения о расширении scope или переносе.
3. **Архитектурное решение, которого нет в ADR.** Нужно выбрать между подходами (библиотека, паттерн), и оба не противоречат существующим ADR, но выбор повлияет на будущее. → Стоп, описать варианты, ждать решения.
4. **Sprint prompt неоднозначен или внутренне противоречив.** → Стоп, конкретный вопрос, не пытайся угадать.

**Не эскалировать:**

- Конкретные технические детали реализации (как именно структурировать функцию, какой regexp использовать)
- Стилистические выборы в рамках project-specific правил CLAUDE.md
- Уточнения по UX-копирайту (если sprint prompt дал общий смысл, можно выбрать слова) — если не уверен, спроси

## Что делать в начале каждой новой сессии

1. **Прочитай `CLAUDE.md` целиком.** Поведенческие правила и anti-patterns обновляются по ходу проекта.
2. **Прочитай sprint prompt текущего спринта.** Это твоя основная задача.
3. **Прочитай файлы, на которые sprint prompt ссылается** (architecture, ADR, предыдущий sprint-report).
4. **Сформулируй краткий план реализации** (1-3 предложения) — что собираешься делать, в каком порядке. Это помогает поймать недопонимание до начала работы.
5. **Начинай кодить.**

В конце:

1. **Запусти проект локально**, проверь что собирается и базовый сценарий работает.
2. **Сделай git commit(ы)** — лучше серия осмысленных коммитов чем один монолитный.
3. **Создай `sprint-report-N.md`** по шаблону.
4. **Сообщи пользователю** что спринт готов к code review.
