# Dev Cycle

> Цикл разработки `<project>`. Двухинстансная схема (Cowork + Claude Code), спринты 2-5 дней работы.

---

## Участники и роли

| Роль | Кто | Что делает |
|---|---|---|
| Product Owner / Designer / QA | Пользователь | Принимает все продуктовые и архитектурные решения. Тестирует на машине. |
| PM / Architect / Documentation | Claude Cowork | Планирование спринтов, формирование ADR совместно с пользователем, вся документация, sprint prompts, code review, тест-кейсы |
| Developer | Claude Code | Пишет код, делает sprint-reports, исправляет баги |

**Принципы:**

- Cowork и Code — **разные инстансы с разным контекстом**. Cowork не трогает код. Code не занимается планированием и архитектурой.
- Cowork **может редактировать** все документы в `docs/` и `CLAUDE.md`. Правки в `docs/context/` (ADR, концепция, архитектура) — только по результатам обсуждения с пользователем, не самостоятельно.
- Пользователь — единственный источник продуктовых решений и единственный, кто видит реальный результат в браузере.

---

## Типы спринтов

Не все спринты одинаковы. Разные типы — разные циклы:

### A. Code sprint (стандартный)

Спринт по разработке фичи или фикса. Полный 9-фазный цикл (см. ниже). Большинство спринтов — этого типа.

**Примеры:** "Sprint 3: страница списка задач с фильтрами", "Sprint 7: форма создания записи с валидацией", "Sprint 12: фикс бага сортировки в таблице".

### B. Content sprint

Спринт без кода — **создание контента** (документация, тексты UI, seed-данные, шаблонные файлы). Урезанный цикл: PLAN → CONTENT → REVIEW → CLOSE. Нет TEST PREP / QA / FIX / RETEST в традиционном виде.

**Примеры:**
- Написание текстов tour-плашек / онбординга
- Подготовка пользовательской документации
- Подготовка seed-данных и текстов для пустых состояний UI

**Особенности:**
- "Тестирование" = ревью смысла. Пользователь читает, оценивает: понятно? полезно? правдоподобно?
- Вместо sprint-report-N.md в традиционном виде — короткий `content-N-review.md` с заметками автора.
- Закрытие на фазе CLOSE — обновление JTBD и Timeline в CONTEXT.

### C. Architecture sprint

Спринт с **архитектурной работой до начала кода**. Нужен когда задача требует решений по технологиям/паттернам, которых ещё нет в ADR.

**Примеры:**
- Дизайн схемы хранения данных перед первой фичей, использующей это хранение
- Выбор библиотеки (виз / парсер / валидатор) с разбором альтернатив
- Финализация схемы YAML frontmatter перед фичей чтения/записи

**Особенности цикла:**
- Фаза PLAN расширена: включает **research + проектирование** (живёт в `docs/project/sprint-N-design.md`)
- Перед DEV фазой — **review архитектурного решения** пользователем
- Если решение значимое и долгоживущее — Cowork в диалоге с пользователем формирует новый ADR в `docs/context/decisions-log.md`
- Дальше — обычный Code sprint по реализации согласованного дизайна

---

## Связь скоупа и спринтов

`docs/project/JTBD.md` — это **полный backlog**, не дисциплинирующая граница спринта. Скоуп каждого конкретного спринта формируется в фазе PLAN из подмножества user stories.

**Правило:** брать в спринт то, что осмысленно сделать за 2-5 дней работы. Если фича большая — она разбивается на N спринтов.

Cowork при планировании каждого следующего спринта смотрит **что фактически сделано в JTBD.md**, не "пора уже взяться за блок X".

## Цикл Code sprint (стандартный)

Каждый Code sprint проходит через 9 фаз. Фазы 6-8 могут повторяться (цикл фиксов), но это **штатная часть процесса**, не аварийный спринт.

```
  ┌──────────────────────────────────────────────────────────────────────┐
  │                          SPRINT N                                    │
  │                                                                      │
  │  1. PLAN ──► 2. PROMPT ──► 3. DEV ──► 4. CODE REVIEW ──► 5. TEST PREP│
  │                                                              │       │
  │                                                              ▼       │
  │  9. CLOSE ◄── 8. RETEST ◄── 7. FIX ◄────────────────── 6. QA        │
  │       │            │                                                 │
  │       │            └─── (если есть новые баги) ────► 7. FIX          │
  │       │                                                              │
  │       └──► SPRINT N+1                                                │
  └──────────────────────────────────────────────────────────────────────┘
```

**Краткая суть фаз:**

- **PLAN, PROMPT** — Cowork готовит задачу
- **DEV** — Code пишет
- **CODE REVIEW** — Cowork проверяет код против ADR/scope/architecture, формирует список concerns
- **TEST PREP** — Cowork готовит тест-кейсы (с учётом результатов code review)
- **QA** — пользователь тестирует в браузере
- **FIX, RETEST** — устранение багов (циклически, до 2 итераций)
- **CLOSE** — обновление документации, закрытие спринта

---

### Фаза 1: PLAN (Cowork + Пользователь)

**Цель:** определить что делаем в спринте и зафиксировать scope.

**Действия:**

- Обсудить текущее состояние проекта (что готово, что блокирует) — Cowork читает последние sprint-reports и JTBD
- Выбрать scope из `docs/project/JTBD.md`
- Определить цель спринта одним предложением
- **Проверить scope против ADR** — не противоречит ли цель архитектурным решениям? Если есть конфликт — стоп, обсуждаем с пользователем (возможно нужен новый ADR)
- Обсудить риски и неясности до начала разработки
- Если есть UX-решения — принять их **до** передачи в разработку

**Артефакт:** обсуждение в Cowork. Финальный scope фиксируется в sprint prompt (Фаза 2).

**Правила:**

- Scope должен быть реалистичным. Лучше сделать меньше и закрыть, чем растянуть спринт.
- Если в процессе обсуждения всплывают вопросы, требующие исследования — это **Architecture sprint**, не часть планирования Code sprint.
- Пользователь имеет право вето на любую фичу в scope.

**Анти-паттерн:** Cowork **не должен** сам решать "это можно отложить, для MVP не критично" в одностороннем порядке. Если есть подозрение, что фича должна быть в скоупе но "сложная" — обсуждать **до** урезания, не после.

---

### Фаза 2: PROMPT (Cowork)

**Цель:** подготовить детальное задание для Claude Code.

**Действия:**

- Написать `docs/project/sprint-N-prompt.md` — полное ТЗ для Claude Code
- Включить: цель, список задач, технические детали, ограничения (DO NOT), критерии приёмки
- Сослаться на нужные файлы (`CLAUDE.md`, предыдущие sprint-reports, релевантные ADR, разделы architecture)

**Артефакт:** `docs/project/sprint-N-prompt.md`

**Шаблон prompt-файла:**

```markdown
# Sprint N — [Название]

## Overview
[Цель спринта одним абзацем]

## Scope (user stories)
[Список user stories из JTBD которые закрываем, с тегами M если применимо]

## Tasks
[Детальный список задач с техническими деталями]

## Technical Notes
[Архитектурные решения, зависимости, ссылки на документы]

## ADR Constraints
[Какие ADR применимы к этому спринту, что нельзя нарушать]

## Files involved
[Какие файлы создаются / модифицируются — для navigation]

## Acceptance criteria
[Что должно работать end-to-end для приёмки. Smoke-test.]

## DO NOT
[Что НЕ трогать. Соседние модули, не относящиеся к скоупу.]
```

**Правила:**

- Prompt должен быть **самодостаточным** — Claude Code не читает историю чатов из Cowork, только файлы.
- Чем детальнее prompt, тем лучше результат. Не жалеть времени на эту фазу.
- Если есть UX-решение — описать его в prompt или сослаться на файл/скриншот.
- **Обязательно** упомянуть какие ADR применимы (ссылки на `docs/context/decisions-log.md`).
- Секция **DO NOT** обязательна, даже если кажется очевидной. Перечислить что **рядом**, но **не трогаем**.

### Фаза 3: DEV (Claude Code)

**Цель:** реализация задач спринта.

**Действия:**

- Пользователь запускает Claude Code, направляет на `sprint-N-prompt.md`
- Claude Code пишет код, делает коммиты
- По завершении формирует отчёт

**Артефакт:** `docs/project/sprint-report-N.md`

**Шаблон отчёта** (Claude Code должен следовать этому формату):

```markdown
# Sprint N Report — [Название]

**Dates:** YYYY-MM-DD
**Status:** Complete / Partial

## Goal
[Цель спринта одним предложением]

## What was built
[Описание реализованного с деталями. Каждая user story из scope — отдельный пункт.]

## Files Created
| File | Purpose |
|------|---------|

## Files Modified
| File | Changes |
|------|---------|

## ADR Compliance
[Какие ADR были применимы и как они соблюдены. Если что-то нарушено — обоснование почему.]

## Known Issues
[Что не удалось, ограничения, технический долг]

## Notes
[Дополнительные наблюдения, вопросы для обсуждения с Cowork/пользователем]
```

**Правила:**

- Пользователь может задавать вопросы Claude Code во время разработки, но **крупные изменения scope** — через Cowork.
- Если Claude Code сталкивается с проблемой, которая меняет scope — **остановиться**, вернуться к обсуждению через Cowork.
- Если Claude Code сталкивается с противоречием ADR — **остановиться**, не пытаться "обойти" молча.

---

### Фаза 4: CODE REVIEW (Cowork)

**Цель:** независимая проверка кода другим LLM-инстансом до того как пользователь начнёт QA.

**Зачем это нужно:** Code и Cowork — разные инстансы с **разным контекстом**. Cowork знает все ADR, видит sprint prompt, читал прошлые reports. Code писал код фокусируясь на задаче. Cowork ловит то что Code пропустил из-за scope-tunnel-vision: нарушения ADR, скоуп-крип, несогласованность с архитектурой, нарушения anti-patterns.

**Что Cowork делает:**

- Читает `sprint-N-prompt.md`, `sprint-report-N.md`, git diff кода спринта
- Проверяет:
  - **ADR compliance** — нет ли нарушений зафиксированных решений
  - **Scope** — каждое изменение трассируется к sprint prompt? Нет ли лишних правок?
  - **Architecture** — код лёг в правильные модули (по `architecture.md`)?
  - **Anti-patterns** из CLAUDE.md — не появилось ли что-то из списка
  - **Согласованность с прошлыми спринтами** — паттерны не противоречат тому что делали раньше
- Формирует список concerns с категориями (см. ниже)

**Что Cowork НЕ делает:**

- Не предлагает альтернативный код ("я бы написал через reduce, а не через map")
- Не пишет рефакторинг сам
- Не углубляется в стиль кода (это уровень CLAUDE.md правил, проверяет Code сам)
- Не запускает код (это пользователь на QA)

**Артефакт:** `docs/project/code-review-sprint-N.md`

**Шаблон:**

```markdown
# Code Review Sprint N — [Название]

**Reviewer:** Cowork
**Date:** YYYY-MM-DD

## Summary
[1-2 предложения общей оценки]

## Concerns

### 🔴 Blockers (требуют fix до QA)
| # | Где | Что не так | Какой ADR/правило |
|---|-----|-----------|-------------------|
| 1 | `src/lib/parser.js:42` | Парсит YAML через RegExp вместо js-yaml | ADR-003 |

### 🟡 Concerns (обсуждаемые)
| # | Где | В чём concern |
|---|-----|---------------|
| 1 | `src/components/Form.jsx` | 200+ строк в одном компоненте, может стоит разбить |

### 🟢 Notes (наблюдения, не требуют фикса в этом спринте)
| # | Где | Заметка |
|---|-----|---------|
| 1 | `src/lib/storage/` | В будущем стоит добавить retry логику для localStorage quota — сейчас не критично |

## ADR Compliance Check
- ADR-001 (no backend): ✅ нет cloud-зависимостей в diff
- ADR-003 (YAML через js-yaml): 🔴 см. Blocker #1

## Scope Compliance
✅ Все изменения трассируются к задачам в sprint prompt
[или: 🟡 файл X изменён — это в скоупе?]

## Decision Log
[Заполняется после диалога пользователь ↔ Cowork по concerns]
```

**Правила:**

- Cowork **не угадывает** — если что-то не уверен, ставит 🟡 Concern с вопросом, не 🔴 Blocker.
- 🔴 Blocker = объективное нарушение зафиксированного правила (ADR, anti-pattern из CLAUDE.md).
- 🟡 Concern = "подозрительно, обсуди".
- 🟢 Note = "на будущее, не сейчас".

**Дальше — диалог пользователь ↔ Cowork:**

По каждому concern пользователь принимает решение:

- **"Согласен — fix"** → переходит в backlog для FIX phase (как обычные баги)
- **"Не согласен — discard"** → отмечается в Decision Log с обоснованием
- **"Нужно обсудить"** → разговор, итог фиксируется

Спорные моменты могут привести к одному из:

| Что обсудили | Куда фиксировать |
|---|---|
| Архитектурное решение длительного действия | Новый ADR в `docs/context/decisions-log.md` (Cowork формирует по итогам диалога с пользователем) |
| Стандарт кода для всего проекта | `CLAUDE.md`, секция `## Project-specific правила` |
| Анти-паттерн на будущее | `CLAUDE.md`, секция `## Anti-patterns` |
| Project-specific (только этот sprint) | `code-review-sprint-N.md` секция Decision Log |
| Замечание не сделали, но оно общее | `docs/project/CONTEXT.md` (Timeline / Tech debt) |

**Важно:** правки в `docs/context/` (ADR, концепция, архитектура) и в `CLAUDE.md` Cowork делает **только по результатам обсуждения с пользователем**, не самостоятельно. Если есть сомнение — спрашивает.

**Анти-паттерны на этой фазе:**

- ❌ Cowork и Code начинают переписывать друг друга в кругу — модерирует пользователь
- ❌ Cowork предлагает рефакторинг "потому что я бы сделал иначе" — это не его scope
- ❌ Пользователь пропускает Blockers потому что "потом починю" — Blocker блокирует QA до фикса

**Когда фаза заканчивается:**

- Все 🔴 Blockers разрешены (либо fix-нуты, либо переквалифицированы в 🟡/🟢 после обсуждения)
- 🟡 Concerns обсуждены, по каждому есть решение в Decision Log
- 🟢 Notes зафиксированы (если применимо — в tech-debt или CONTEXT)

### Фаза 5: TEST PREP (Cowork)

**Цель:** подготовить тест-кейсы для приёмки.

**Действия:**

- На основе sprint-report, sprint-prompt и code-review результатов создать `docs/project/test-cases-sprint-N.md`
- Включить: позитивные сценарии, негативные сценарии, edge cases, регрессию предыдущих спринтов
- **Особо проверить участки, где были 🟡 Concerns** в code review — там высокая вероятность найти runtime-баги
- Каждый тест-кейс: описание, ожидаемый результат, поле для статуса

**Артефакт:** `docs/project/test-cases-sprint-N.md`

**Шаблон тест-кейса:**

```markdown
# Test Cases Sprint N — [Название]

## Test cases

| # | Test Case | Expected | Status |
|---|-----------|----------|--------|
| 1 | [Описание действия] | [Что должно произойти] | ⬜ |

Статусы: `⬜` (не проверен), `ok` (пройден), `bug` (баг, описать в таблице ниже).

## Bugs found

| ID | Severity | Test case # | Description | Steps to reproduce |
|----|----------|-------------|-------------|-------------------|
```

**Правила:**

- Тест-кейсы пишутся **до** начала тестирования, не параллельно.
- Регрессионные тесты обязательны (ключевые сценарии предыдущих спринтов).
- Количество тест-кейсов разумное — 15-30 на спринт, не 100.

**Что проверять в web-проекте (общий чек-лист):**

- Позитивные сценарии user stories из scope
- Edge cases ввода (пустые поля, очень длинные значения, специальные символы)
- Состояния загрузки и ошибок
- Адаптивность (если есть mobile/desktop)
- Browser compatibility (минимум Chrome + Safari + Firefox если в скоупе)
- Сохранение состояния (localStorage / sessionStorage / cookies) при перезагрузке
- A11y базово: keyboard navigation для ключевых сценариев, alt-тексты для важных картинок
- Регрессия: ключевые сценарии 2-3 предыдущих спринтов

---

### Фаза 6: QA (Пользователь)

**Цель:** протестировать в браузере и зафиксировать результаты.

**Действия:**

- Запустить локально (`npm run dev`, `pnpm dev` — что есть в проекте) или собрать preview-сборку
- Пройти по тест-кейсам, отмечая статусы
- Найденные баги — описать в таблице багов с severity
- По возможности — скриншоты или описание шагов воспроизведения

**Артефакт:** обновлённый `docs/project/test-cases-sprint-N.md` с заполненными статусами и таблицей багов.

**Severity:**

- **Critical:** приложение крашится, теряет данные пользователя, или ключевой user story не работает вообще
- **High:** основная фича спринта работает с явными нарушениями (неправильные данные, неработающие кнопки)
- **Medium:** фича работает, но с заметными проблемами (визуальные баги, неудобства)
- **Low:** косметика, мелкие неудобства

---

### Фаза 7: FIX (Cowork + Claude Code) — если есть баги

**Цель:** исправить найденные баги.

**Действия:**

- Обсуждение результатов QA в Cowork
- Классификация багов: fix now (Critical/High) vs defer (Medium/Low)
- Подготовка fix prompt: `docs/project/sprint-N-fix-prompt.md`
- Claude Code выполняет фикс
- Claude Code обновляет `sprint-report-N.md` (добавляет секцию "Bug Fixes") или создаёт `sprint-N-fix-report.md` (по договорённости — для большого спринта удобнее отдельный файл)

**Артефакт:** `docs/project/sprint-N-fix-prompt.md`, обновлённый `sprint-report-N.md` (или `sprint-N-fix-report.md`)

**Шаблон fix prompt:**

```markdown
# Sprint N Fix Prompt — [короткое описание итерации]

## Context
[Откуда взялись эти баги: ссылка на test-cases-sprint-N.md, какая итерация фиксов]

## Bugs to fix

### Bug #1 — [Severity, короткое название]
**Description:** [что не так]
**Steps to reproduce:** [шаги]
**Expected:** [что должно быть]
**Proposed fix:** [предложение Cowork или "обсудить с пользователем"]
**Files involved:** [где, скорее всего, лежит причина]

### Bug #2 ...

## DO NOT
[Что НЕ трогать. Особенно важно в фазе FIX — велик соблазн "заодно".]
```

**Правила:**

- Это **не отдельный спринт**. Это штатная фаза текущего спринта. Не нумеровать как "Sprint N.5".
- Fix prompt должен содержать: ID бага, описание проблемы, предлагаемый фикс, что НЕ трогать.
- Deferred баги переносятся в `JTBD.md` (отдельной секцией или в исходную user story) с пометкой из какого спринта.

---

### Фаза 8: RETEST (Пользователь) — если были фиксы

**Цель:** проверить что фиксы работают и ничего не сломалось.

**Действия:**

- Перепроверить тест-кейсы с багами (те, что были `bug`)
- Пройти ключевые регрессионные тесты
- Обновить статусы в test-cases

**Артефакт:** обновлённый `docs/project/test-cases-sprint-N.md`

**Правила:**

- Если после фикса появились новые баги — вернуться в Фазу 7, но **не более 2 итераций**. Если после 2 итераций остаются баги — deferred в следующий спринт, текущий закрывается с пометкой Partial.
- Финальный вердикт за пользователем: "принято" или "нужна ещё итерация".

---

### Фаза 9: CLOSE (Cowork)

**Цель:** закрыть спринт и обновить документацию.

**Действия:**

- **Обновить `docs/project/JTBD.md`:** отметить `[x]` завершённые user stories, проставить пометки если применимо
- **Обновить `CLAUDE.md`** (если изменилось — например добавлены новые anti-patterns или project-specific правила)
- **Обновить `docs/project/CONTEXT.md`:** добавить запись в Development Timeline (что закрыто, что узнали, что переносится)
- Deferred баги — зафиксировать (куда перенесены)
- Краткое ретро: что сработало, что нет, что улучшить в процессе

**Артефакты:** обновлённые `JTBD.md`, `CLAUDE.md` (при необходимости), `CONTEXT.md`.

**Правила:**

- Спринт **не закрывается** пока документация не обновлена. Это обязательный шаг.
- Ретро не обязано быть длинным — 3-5 предложений достаточно.
- Если в спринте всплыли архитектурные вопросы — обсудить с пользователем, по итогу Cowork формирует новый ADR в `decisions-log.md`.

## Цикл Content sprint (урезанный)

Для спринтов без кода — упрощённый процесс:

### Фаза 1: PLAN
То же что Code sprint — выбор скоупа, цель.

### Фаза 2: CONTENT (Cowork или Пользователь)
Создание контента. Может быть совместно — Cowork делает черновик, пользователь дорабатывает.

**Артефакт:** контент-файлы в нужных местах проекта.

### Фаза 3: REVIEW (Пользователь + опционально Cowork)
Пользователь читает результат, оценивает:

- Понятно ли?
- Реалистично/полезно ли?
- Соответствует ли формату?

**Артефакт:** короткий `docs/project/content-N-review.md` с заметками "что хорошо, что переписать".

### Фаза 4: CLOSE
Обновление JTBD и CONTEXT, как в Code sprint.

**Особенность Content sprints:** контент часто требует **2-3 итераций**. Не "написал → принял", а "черновик → ревью → правки → финал".

## Цикл Architecture sprint (с research-фазой)

Для спринтов, где требуется архитектурная работа до кода:

### Фаза 1: PLAN
То же. Цель = "решить архитектурный вопрос X и подготовить scope для следующего Code sprint".

### Фаза 2: RESEARCH + DESIGN (Cowork)

- Изучение вариантов (через web search, изучение docs библиотек)
- Подготовка `docs/project/sprint-N-design.md` с обоснованием выбора
- Возможные альтернативы с trade-off'ами
- Рекомендация

**Артефакт:** `docs/project/sprint-N-design.md`

### Фаза 3: REVIEW (Пользователь + Cowork)

- Пользователь оценивает рекомендацию
- Если решение значимое и долгоживущее — Cowork по итогам диалога формирует новый ADR в `docs/context/decisions-log.md`

**Артефакт:** утверждённый design doc + опционально новый ADR.

### Фаза 4: HAND-OFF
Architecture sprint **завершается** передачей в следующий Code sprint. Design doc становится частью sprint prompt'а следующего спринта.

**Особенность:** Architecture sprint **не пишет код**. Если в его рамках написан какой-то PoC — это отмечается отдельно как "experimental, не идёт в main".

## Правила для документации

1. **`docs/project/JTBD.md`** — обновляется на фазе CLOSE каждого спринта. Чек-боксы отражают реальное состояние.
2. **`CLAUDE.md`** — источник правды для **обоих** инстансов. Обновляется когда появляются новые anti-patterns или изменяются project-specific правила.
3. **`docs/project/CONTEXT.md`** — история проекта. Каждый закрытый спринт — новая запись в Timeline.
4. **`docs/context/`** — стабильные документы (концепция, архитектура, ADR). Cowork **может редактировать**, но только по результатам обсуждения с пользователем. Самостоятельные правки — нет.
5. **Sprint report** — пишется Claude Code, формат стандартизирован (шаблон выше).
6. **Test cases** — пишутся Cowork **до** тестирования. Заполняются пользователем **после**.

## Чеклист смены фаз (Code sprint)

Переход из фазы в фазу — через явное подтверждение в Cowork.

| Переход | Что нужно проверить |
|---|---|
| PLAN → PROMPT | Scope согласован, UX-вопросы закрыты, ADR-конфликтов нет |
| PROMPT → DEV | Prompt файл создан и просмотрен пользователем |
| DEV → CODE REVIEW | Sprint report получен от Claude Code |
| CODE REVIEW → TEST PREP | Все 🔴 Blockers разрешены, 🟡 Concerns обсуждены, Decision Log заполнен |
| TEST PREP → QA | Test cases файл создан |
| QA → FIX | Test cases заполнены, баги классифицированы |
| FIX → RETEST | Fix report получен, код обновлён |
| RETEST → CLOSE | Все Critical/High баги закрыты или осознанно deferred |
| CLOSE → следующий PLAN | Документация обновлена (JTBD, CONTEXT, при необходимости CLAUDE.md) |

## Калибровка Cowork — типичные ошибки

> Универсальные паттерны, которые я (Cowork) должен ловить у себя. По мере накопления project-specific ошибок — добавлять в `CLAUDE.md` секцию `## Anti-patterns`.

### Ошибка 1: Подмена продуктовой логики аргументами "для демо/MVP не критично"

**Симптом:** "Эта фича не нужна для базового сценария → отложим из MVP".

**Почему ошибка:** demo/MVP-сценарий — это упрощённая картинка. **Продукт должен работать в реальных условиях с реальной аудиторией.** Когда автор называет продуктовую боль из опыта — она не отменяется аргументом "для демо не критично".

**Как избегать:** при выборе scope MVP приоритет = что нужно пользователю, не что красивее в демо/видео. Если есть подозрение что фича "сложная и можно отложить" — обсуждать **до** урезания, не после.

### Ошибка 2: Преждевременное схлопывание архитектурных решений

**Симптом:** "Запретить класс X" (no backend, no auth, no third-party libs) без разложения на под-категории.

**Почему ошибка:** широкий запрет съедает кейсы, которых не подумали. "No auth" может съесть OAuth для интеграций, basic auth для preview-сборки, и login для будущего pro-плана — а имелось в виду только последнее.

**Как избегать:** когда видишь решение "запретить класс вещей", разложи класс на под-категории и явно скажи что запрещено, что — нет.

### Ошибка 3: Скоуп-крип в фазе DEV

**Симптом:** Claude Code "заодно отрефакторит соседний модуль", "улучшит соседний UI", "добавит ещё одну мелочь".

**Почему ошибка:** прямо нарушает Surgical Changes из CLAUDE.md. Каждое изменение должно трассироваться к sprint prompt.

**Как избегать:** sprint prompt должен явно содержать секцию "DO NOT" с перечислением того, что **рядом**, но **не трогаем**.

### Ошибка 4: Пропуск ADR при планировании

**Симптом:** scope спринта противоречит ADR, и Cowork не заметил при PLAN.

**Почему ошибка:** Claude Code будет либо нарушать ADR, либо тратить время на эскалацию.

**Как избегать:** в фазе PLAN **обязательно** проверить — не противоречит ли цель спринта какому-то ADR. Если противоречит — это либо ошибка планирования, либо повод для нового ADR (обсуждается с пользователем, фиксируется Cowork).

### Ошибка 5: Code review с альтернативными реализациями

**Симптом:** в code review Cowork предлагает "я бы написал не через map а через reduce", "лучше использовать другой паттерн".

**Почему ошибка:** это не scope CODE REVIEW phase. Cowork **только flag concerns**: ADR compliance, scope, architecture, anti-patterns. Стиль кода — зона CLAUDE.md правил, проверяет Code сам.

**Как избегать:** перед каждым пунктом code review спроси — это нарушение зафиксированного правила (ADR, anti-pattern из CLAUDE.md), или это моё стилистическое мнение? Если второе — не пиши.

### Ошибка 6: Скоуп-крип в фазе FIX

**Симптом:** в fix prompt появляются "заодно" пункты, которые не были багами в test-cases.

**Почему ошибка:** FIX phase должна закрывать конкретные баги из QA, не превращаться в мини-спринт улучшений.

**Как избегать:** каждый пункт в fix prompt должен иметь ID бага из test-cases-sprint-N.md. Если что-то не в test-cases, но хочется починить — это в JTBD как новую user story для следующего спринта.
