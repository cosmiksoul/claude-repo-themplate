# Decisions Log — `<project>`

> Architecture Decision Records. Каждое значимое архитектурное или продуктовое решение фиксируется здесь.
>
> **Кто пишет:** Cowork по результатам обсуждения с пользователем. Самостоятельно — нет.
>
> **Когда писать ADR:** решение долгоживущее (повлияет на множество будущих спринтов), архитектурное (стек, паттерн, ограничение), или продуктовое стратегическое (что НЕ делаем, почему).

---

## Формат записи

```markdown
## ADR-NNN — [Короткое название решения]

**Date:** YYYY-MM-DD
**Status:** Accepted / Superseded by ADR-MMM / Withdrawn

**Context:**
[Что заставило принять решение. Какая проблема, какие альтернативы рассматривались.]

**Decision:**
[Что решили. Конкретно, проверяемо.]

**Consequences:**
[Что теперь нельзя или нужно делать иначе. Что это разблокирует. Что закрывает.]

**Alternatives considered:**
- [Альтернатива 1] — отклонено потому что ...
- [Альтернатива 2] — отклонено потому что ...

**Related:**
[Ссылки на другие ADR, sprint-reports, обсуждения]
```

---

## Принципы ADR

1. **ADR пишется до реализации**, не после. Если решение принято в фазе CODE REVIEW (что-то всплыло) — фиксируется тогда же, до того как код пойдёт в main.
2. **ADR не отменяется молча.** Если решение пересматривается — новый ADR со статусом, старый помечается `Superseded by ADR-MMM` или `Withdrawn`.
3. **ADR содержит обоснование.** "Так решили" — недостаточно. Должно быть видно почему.
4. **ADR — не догма.** Если по ходу проекта стало понятно, что ADR ошибочный — открыто переcмотреть, не "обходить молча".
